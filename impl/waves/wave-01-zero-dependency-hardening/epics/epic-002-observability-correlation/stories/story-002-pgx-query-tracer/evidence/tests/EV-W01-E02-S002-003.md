# EV-W01-E02-S002-003 — Literal-leakage negative test

## Evidence ID

## Story and task

## Acceptance criteria proven

AC-W01-E02-S002-02 (literal-leakage portion: "the statement-summary attr contains no literal
parameter values from a parameterized query").

## Execution command

`DATABASE_URL='postgres://wowapi:wowapi-local-only@localhost:5432/wowapi?sslmode=disable' go test ./kernel/database/ -run 'TestIntegrationQueryTracerDoesNotLeakBoundParameterLiterals' -v -race -count=1`

Executes `SELECT $1::text` with a sensitive-looking bound value
(`hunter2-super-secret-literal`), then scans EVERY string attribute of EVERY exported span for the
literal, and additionally asserts `db.statement` contains the `$1` placeholder (i.e. the
parameterized text as issued, not rendered SQL). The implementation never touches
`TraceQueryStartData.Args`, so bound values cannot reach attrs by construction; this test guards
that property behaviorally.

## Result

**Pass.** No exported span attribute contains the bound literal; `db.statement` carries the
placeholder form. Residual risk (recorded, unchanged from plan): SQL assembled by string
concatenation would surface its embedded literals in `db.statement` — inherent to tracing SQL text
as issued, flagged in `WithQueryTracer`'s doc comment, and no wowapi call path does this.
