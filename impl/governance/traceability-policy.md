# Traceability policy

Mandate §2.4. Traceability is what lets anyone determine, for any source requirement, whether it
was implemented, deferred, rejected, or superseded — without re-reading the original planning
documents.

## The chain (mandate §2.4, verbatim)

```text
Source document
→ Source requirement or finding
→ Wave
→ Epic
→ Story
→ Acceptance criterion
→ Task
→ Implementation change
→ Artifact
→ Verification evidence
→ Review
→ Acceptance
```

Every link in this chain must be reconstructable by following front-matter fields and file
content — never by tribal knowledge. A story that cannot show its `source_requirements`, its
`acceptance_criteria`, and (once implemented) its tasks/artifacts/evidence/review/acceptance
trail is not traceable regardless of whether the code itself is correct.

## How this repo's matrices are derived

`impl/analysis/requirement-inventory.md` is the canonical allocation (its own front matter:
`derived: false`) — every material item, classified, dispositioned, and assigned a target
wave/epic/story. It is authored directly from the source documents, not generated.

Everything downstream of it in `impl/tracking/` is a **derived view**, generated by walking front
matter, never hand-maintained as a second source of truth:

- **`source-traceability-matrix.md`** is generated by walking every item's `source_requirements`
  front-matter field (on `story.md`, and transitively `epic.md`/`wave.md`) back to its row in
  `impl/analysis/requirement-inventory.md`. This answers: for a given source document/section,
  what requirement did it produce, and what is its disposition and target?

- **`requirement-traceability-matrix.md`** is generated by walking each
  `requirement-inventory.md` row **forward**: the row's `Target` column already names the
  wave/epic/story assignment; from there the matrix follows into that story's
  `acceptance_criteria` front-matter list, then (once populated) into `depends_on`/task
  references, `artifacts`, and `evidence` front-matter arrays. This answers: for a given
  requirement, what wave/epic/story implements it, what AC proves it, and what's the final
  result?

Both matrices are mechanical rollups of front matter that already exists on the canonical files.
Regenerating a matrix must never require inventing information not present in
`requirement-inventory.md` or the item's own front matter.

## Canonical data lives in the items themselves

Per mandate §6 (also stated in `status-model.md` and `impl/index.md` "Progress maintenance"):

> Roll-up indexes and registers should either be generated from metadata or clearly marked as
> derived views.

`source-traceability-matrix.md` and `requirement-traceability-matrix.md` are both derived views.
If either ever conflicts with a story's own front matter or with
`impl/analysis/requirement-inventory.md`, the canonical file wins and the matrix is stale and
must be regenerated — the matrix is never edited directly to "fix" a discrepancy.

## What traceability must be able to prove

Mandate §17 validation checklist items 4, 5, 6, 7, 9, and 10 — the specific claims this
traceability structure must be able to substantiate on demand:

4. Every planned requirement maps to a wave, epic, and story.
5. Every story has measurable acceptance criteria.
6. Every story has implementation, verification, artifact, evidence, deviation, and closure
   structures.
7. Dependencies are explicitly recorded.
9. Duplicated requirements are consolidated (see `impl/analysis/duplicate-analysis.md`).
10. Conflicts are documented and resolved or escalated (see
    `impl/analysis/conflict-resolution.md`).

A traceability review that cannot answer all six of these for a given requirement or story has
found a gap in the programme structure, not merely a documentation nit — it must be corrected by
fixing the canonical item's front matter/content, then regenerating the derived matrices, not by
hand-patching a matrix row.
